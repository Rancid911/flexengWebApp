Status legend:

- `healthy`: current separation is good enough; only incremental cleanup is needed.

- `needs_split`: known module or route has mixed responsibilities and should be split in a planned refactor.

## Current `needs_split` Targets

- None currently tracked as active domain-level `needs_split`.
- The active architecture backlog is empty and the broad migration is complete. Future `needs_split` targets should be added only when a concrete issue, feature, or review finding exposes a real boundary problem.
- `scripts/check-architecture.mjs` protects `app/` as routing-only, so future feature/client/server implementation files should start in `features/*`, `shared/*`, or `lib/*` instead of creating new route-layer backlog.
- Protected API routes must keep using `requirePermission()`. Public, provider-authenticated, and internal utility endpoints stay outside permissions only through the explicit architecture allowlist.

## Ownership Rule

When adding a feature, first decide its domain. If it is a read-only aggregate across domains, label it as an aggregator. If it mutates data, the mutation should live in the owning domain service.
